tuiparts.sh has two product layers with different ownership and update models.
Primitives centralize reusable terminal interaction. Recipes keep opinionated
presentation inside the application that renders it.

## The two layers

| Layer | Owns | Distribution | Update model |
| --- | --- | --- | --- |
| Primitive packages | State, interaction, focus, collections, overlays, lifecycle, and public Parts | npm packages | Update with the package manager |
| Registry Recipes | Layout, colors, glyphs, labels, variants, and convenience assembly | Copied TypeScript source | Review and merge with shadcn diff |

OpenTUI remains the platform layer. When OpenTUI already owns a behavior, a
Primitive preserves that contract instead of building a parallel state
machine. Input is the clearest example: it retains OpenTUI's mutable buffer and
event order while the Recipe adds visual defaults.

## Primitive ownership

A Primitive owns only behavior that must remain consistent across
presentations. Depending on its interaction model, that can include:

- controlled and uncontrolled state;
- keyboard, pointer, and imperative activation;
- focus eligibility and restoration;
- collection registration and roving focus;
- overlay stacking, containment, and dismissal;
- readonly public state and semantic change details;
- cleanup when a behavior-bearing node is destroyed.

It does not choose colors, spacing, glyphs, labels, semantic visual variants,
or a fixed visual tree beyond the Parts needed for behavior.

## Roots, Parts, and Stores

A **Root** is the ownership and composition boundary for one Primitive
instance. It owns state directly or attaches to a Store when several
Renderables must coordinate.

A **Part** is a public Renderable with a behavior or semantic responsibility.
For example, `Switch.Root` owns checked state while `Switch.Thumb` reads that
state and leaves its position and content to the caller.

A **Store** is the framework-neutral owner of state or cross-Renderable
coordination. Core callers wire Stores explicitly. React and Solid adapters
normally create and provide them, so framework consumers configure Roots with
props rather than passing Stores through their trees.

## State and refs

Framework render callbacks receive readonly state such as `checked`,
`pressed`, `focused`, or `disabled`. State snapshots are replaced only when
observable behavior changes; callers do not mutate them.

A framework `ref` resolves to the actual Core Renderable for that Part. The
same imperative actions used by Core—such as `press()`—remain available
without a wrapper handle.

## Runtime adapters

`@tuiparts/core` contains framework-neutral Stores and Renderables.
`@tuiparts/react` and `@tuiparts/solid` expose equivalent compound Parts and
adapt their runtime's state flow to the same Core behavior.

The adapters share public vocabulary where runtime semantics permit it, but
their syntax is not identical to imperative Core. Registry reference pages
therefore show framework usage separately from Core factory usage.

## Collections and overlays

Collection Roots such as Radio Group and Toggle Group coordinate item
registration, availability, orientation, and roving focus. Radio always
belongs to Radio Group; Toggle can operate independently or adopt Toggle Group
ownership.

Dialog coordinates renderer-scoped layers. Only the topmost eligible layer
handles Escape or backdrop dismissal, focus remains within the active popup,
and closing restores focus to the appropriate trigger or parent layer.

## Releases

`@tuiparts/core`, `@tuiparts/react`, and `@tuiparts/solid` form the linked
foundation release line and version together. `@tuiparts/dialog` and
`@tuiparts/toast` are independently versioned companion products; they are
not part of that foundation line.

The complete ownership, event, Part, lifecycle, and conformance rules are in

The accepted decisions behind the architecture live in
