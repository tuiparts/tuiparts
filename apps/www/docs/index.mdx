tuiparts.sh provides interaction Primitives and editable UI Recipes for
[OpenTUI](https://github.com/anomalyco/opentui). It supports React, Solid, and
imperative Core applications.

The project separates terminal UI work by ownership:

- **Primitives package behavior.** State, focus, keyboard and pointer
  interaction, collections, overlays, and lifecycle ship in versioned npm
  packages.
- **Recipes copy presentation.** Layout, colors, glyphs, labels, variants, and
  convenience APIs install as TypeScript source owned by your application.

Package the behavior you want to share and test centrally. Copy the
presentation you expect to adapt.

Most users should start with a Recipe. Use a Primitive directly when you need
its behavior without adopting the Recipe's composition or public API.

## Recipes and the Catalog

A Recipe assembles Primitives into useful presentation and installs from the
[Catalog](/catalog) through the shadcn CLI. The installed file is ordinary
Core, React, or Solid source in `components/ui`.

Recipes share a consumer-owned [theme file](/theming), but there is no hidden
styling engine or packaged theme runtime. Change the installed source when
you need a different layout, token mapping, glyph, or public convenience prop.

Some Recipes have no Primitive behind them. Badge, for example, is only
presentation, so it is composed directly from OpenTUI elements and distributed
only as editable source.

## Primitives and Parts

A Primitive is a visually unstyled behavior unit. `@tuiparts/core` provides
framework-neutral Stores and Renderables; `@tuiparts/react` and
`@tuiparts/solid` adapt the same behavior to their runtimes.

Primitives expose independently composable **Parts** where the interaction
model needs them. For example, Checkbox exposes `Root` and `Indicator`.
`Root` owns checked state and activation while `Indicator` reflects that
state without choosing a mark, color, or layout.

Parts are real OpenTUI nodes, not styling slots. You can arrange them, pass
native properties, attach Renderable refs, and supply your own content.

## Choose a layer

| Start with | Use it when | You own |
| --- | --- | --- |
| A Catalog Recipe | You want a working control to use, customize, or reshape into your house style. | The installed source and future merges |
| A packaged Primitive | You need tested behavior without adopting a Recipe's composition or public API. | Presentation and composition |

You can start with a Recipe and later take it apart. The underlying Primitive
remains available when the Recipe's structure no longer fits.

## One model, three runtimes

React and Solid expose compound-part APIs over the same Core behavior. Core
callers construct Renderables and wire Stores explicitly where coordination
requires it. Runtime syntax differs, but ownership, state transitions, and
interaction rules remain aligned.
