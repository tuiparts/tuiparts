The tuiparts.sh Registry serves editable Recipe source as static JSON payloads.
The official shadcn CLI installs that source and its declared dependencies
into your application.

There is no tuiparts-specific installer, Recipe lockfile, or automatic merge
engine. Installed files belong to the consuming project.

## Item addresses

Runtime-specific Recipes use `<runtime>/<name>` addresses:

```text
core/checkbox
react/checkbox
solid/checkbox
```

The official shadcn Registry Directory resolves addresses such as
`@tuiparts/react/checkbox`. The equivalent direct URL is
`https://tuiparts.sh/r/react/checkbox.json`.

Theme presets are framework-neutral and omit the runtime segment, for example
`@tuiparts/theme-catppuccin`.

## Install an item

Install by namespace:

```bash
pnpm dlx shadcn@4.13.0 add @tuiparts/react/checkbox
```

Or install the same item by URL:

```bash
pnpm dlx shadcn@4.13.0 add https://tuiparts.sh/r/react/checkbox.json
```

The CLI copies the Recipe into `components/ui` and installs the dependencies
declared by the item. Every Recipe also declares its runtime's Theme item, so
the first installation adds `components/ui/theme.ts`; React and Solid also
receive `components/ui/use-theme.tsx`.

A fresh OpenTUI project needs no `components.json` or shadcn framework
initialization because each item declares its file targets.

Running an ordinary `add` against an existing file prompts before replacing
it and preserves the local file by default.

## Compatibility metadata

Registry items assume an existing OpenTUI application for their selected
runtime. Interactive Recipes declare only their direct `@tuiparts/core`,
`@tuiparts/react`, or `@tuiparts/solid` package in `dependencies`. The adapter
package owns its Core dependency and checks the application's OpenTUI and
framework versions through peer dependencies.

Registry metadata describes the source lifecycle:

| Field | Meaning |
| --- | --- |
| `type: "registry:item"` | The item installs without shadcn project configuration. |
| `meta.framework` | The Recipe targets Core, React, or Solid. |
| `meta.sourceOwnership: "consumer"` | The installed source is expected to be edited locally. |
| `meta.updateStrategy: "shadcn-add"` | Re-run `add` to check for Recipe updates without replacing local files. |

Presentation-only Recipes such as Badge and Theme add no npm dependencies.
They use the OpenTUI runtime already owned by the application.

## Review updates

Re-run the same `add` command to check the current Recipe. It preserves
existing files unless you explicitly approve or request replacement:

```bash
pnpm dlx shadcn@4.13.0 add @tuiparts/react/checkbox
```

If your project has a valid shadcn `components.json`, you can also inspect
changes before updating:

```bash
pnpm dlx shadcn@4.13.0 add @tuiparts/react/checkbox --view
pnpm dlx shadcn@4.13.0 add @tuiparts/react/checkbox --diff
pnpm dlx shadcn@4.13.0 add @tuiparts/react/checkbox --dry-run
```

Review the output and merge the changes you want. Primitive package updates
arrive through your package manager; Recipe updates are optional source
integrations unless release notes identify a required compatibility change.

:::warning
**`--overwrite` replaces the installed file and discards local edits.** Use it
only when that replacement is intentional.
:::

## Build the Registry

This section is for Registry maintainers. The website build generates every
payload from the repository's `registry.json`:

```bash
pnpm registry:build --output apps/www/public/r
```

Application consumers do not run this command. They install the published
payloads with `shadcn add`.
